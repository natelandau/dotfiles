- When unfamiliar with a tool/platform, research docs before reasoning from general knowledge
- Claude and agents never uses em-dashes and always replaces them with commas, periods, regulard hyphens, or rewrites the sentence.
- Favor branches over worktrees
- Write all plans, specs, and design documents (including any generated by plugins such as superpowers) to a top-level `.agent/` directory in the working repo: design specs in `.agent/specs/`, implementation plans in `.agent/plans/`. Never write these under `docs/`, which is reserved for public-facing documentation.
- Plans, specs, and design documents are scratch working files and must never be committed to git
- When reviewing code or other code review plugins, never disregard a finding on the grounds that it is pre-existing or "not introduced by this diff." We own all the code, so a real problem is in scope regardless of which commit introduced it. Triage review findings by real severity and fixability only. If something is genuinely out of scope for the current change, fix it anyway or flag it explicitly for follow-up. Do not use "pre-existing" as a reason to drop it. Still skip items that are false positives or whose fix would change intended behavior, and say why.
- Pyright is purely a code-navigation aid for agents. It is NOT a project dependency and NOT a binding check. The project's configured typechecker (usually `ty` or `mypy`) is the sole typechecker and is the only tool that should be used to review code. Never worry about or act on Pyright diagnostics.
